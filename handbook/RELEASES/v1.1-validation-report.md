## 4. Automated validation

Command:

```bash
python3 handbook/tools/validate_handbook.py
```

Validated conditions:

- manifest and registry JSON are syntactically valid;
- release versions agree across README, manifest, and registry;
- authority, chapter, profile, template, and route paths exist;
- chapter identifiers, numbers, topics, and rule identifiers are unique;
- chapter discovery and reading order come from the manifest rather than a hard-coded count;
- every numbered chapter contains required metadata and sections 1 through 9 in order;
- References is accepted as the optional tenth section;
- all local Markdown links, rule sources, and heading anchors resolve;
- task routes reference known profiles and chapters;
- rule owners and source files agree;
- exact normative statements are not duplicated across Markdown files;
- fenced code blocks identify their language and close correctly;
- handbook Markdown remains English and contains no unresolved drafting marker.

Final result: Pass.

## 5. Semantic review

### Single source of truth

- The Client Data Policy owns client-data location and disclosure rules.
- The Git Standards own Git history and protected integration controls.
- Repository Standards define capabilities; Repository Template provides a non-normative scaffold.
- Profiles select and order authoritative rules without creating a second normative source.
- The rule registry identifies sources without replacing their text.

### Founder-led operation

- Lightweight work uses compact evidence and explicit self-review.
- Standard work supports founder-held roles plus automated and AI-assisted evidence.
- Controlled work may progress internally while reversible, but requires qualified second-human review before external reliance.
- A qualified client reviewer or contracted specialist may provide the required independent human scope.

### Business-plan alignment

- Active client delivery and statistical evidence now precede generic repository rollout.
- The client profile covers framing, data contract, design, build, verification, statistical validation, packaging, client-side execution, acceptance, and closure.
- Tool-and-data isolation remains the default delivery architecture.
- Product and open-source standards remain available without becoming the first operating priority.

### Team evolution

- Role-based governance remains stable from one to twenty people.
- Additional administrators, reviewers, ownership backup, onboarding, incident control, and cross-project stewardship activate progressively.
- Growth alone does not require a handbook redesign.

### AI knowledge system

- AI agents can select task-specific chapters from a machine-readable route.
- Stable rule IDs support durable citations and change-impact tracking.
- Authority remains in human-readable Markdown.
- The validator detects broken machine indexes and exact cross-file normative duplication.

## 6. Repository adoption review

The handbook repository now provides its own governance baseline:

- contribution and security instructions;
- proprietary status;
- repository changelog;
- ignore and editor configuration;
- automated validation workflow;
- deterministic validation with Python standard-library dependencies only.

## 7. Known limitations

- Exact duplicate detection does not prove semantic equivalence; human semantic review remains required.
- External links are not validated by the local tool and require scheduled review.
- GitHub Actions validation becomes externally observable only after the commit is pushed.
- Project-specific contractual, legal, scientific, and security review remains required.
- The rule registry intentionally indexes major owning rules rather than every normative sentence.

## 8. Publication decision

The v1.1.0 document set resolves the material v1.0 audit findings and satisfies the expanded structural, machine-readability, risk-proportionality, client-delivery, founder-operability, and team-evolution requirements. It is suitable as the active WeianData engineering baseline.
